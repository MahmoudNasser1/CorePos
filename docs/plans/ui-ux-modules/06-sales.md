# خطة UI/UX — المبيعات والفواتير (Sales)

**المرتكز:** [00-design-principles.md](./00-design-principles.md) — **وضوح مالي وإجرائي**: رقم فاتورة، عميل، إجمالي، حالة — دائمًا **في نفس الترتيب** في القائمة والتفاصيل.

---

## 1. نظرة عامة

| المسار | محتوى |
|--------|--------|
| `/dashboard/sales` | ملخص/روابط (إن وُجد) |
| `/dashboard/sales/invoices` | قائمة الفواتير |
| `/sales/invoices/[id]`, `/print` | تفصيل + طباعة A4 |
| `/sales/quotations/*` | عروض أسعار |
| `/sales/new` | **بيع عبر فاتورة** (غير POS) — يجب **توضيح** الفرق في الواجهة |
| مرتجعات | `/sales/returns/*` حسب المشروع |

**مكوّنات:** `InvoiceTable`, `InvoiceForm`, `InvoicePrint`, `InvoicePaymentDialog`، إلخ.

---

## 2. فلسفة «احترافي + بسيط»

- **قائمة الفواتير:** أعمدة **قليلة** (رقم، تاريخ، عميل، مبلغ، حالة) — التفاصيل في فتحة الصف.
- **فاتورة = مستند رسمي:** ترويسة واضحة في الطباعة، **ليس** جدولًا عاريًا في A4.
- **دفع جزئي:** **المتبقي** بخط **أوضح** من «المدفوع» — المستخدم يرى الرقم **الحاسم** بسرعة.

---

## 3. تدفقات

1. **قائمة → تفاصيل → طباعة:** تسلسل **بدون** فقدان سياق (رجوع واضح).
2. **إنشاء فاتورة جديدة:** أضف بنودًا → راجع المجاميع → **حفظ/ترحيل** (حسب المنتج) — **زر واحد** لكل خطوة.
3. **تسجيل دفع:** `InvoicePaymentDialog` → نجاح → تحديث الحالة **في الصف** أو بعد refresh.

---

## 4. الأهداف

| # | الهدف | مؤشر |
|---|--------|------|
| G1 | فلترة/بحث | تاريخ/حالة/عميل إن وُجد |
| G2 | تفصيل | شارات + أزرار مرتبة |
| G3 | طباعة A4 | RTL + رأس/ذيل + page breaks |
| G4 | دفع | متبقي واضح + منع double submit |

## 5. التسكات (مفصّلة)

### 5.1 قائمة الفواتير — **إلزامي**

- [ ] **T6.1** **شريط فلاتر:** تاريخ (من-إلى) + حالة + بحث — **صف واحد** على الديسكتوب، انكماش إلى Sheet على الموبايل.
- [ ] **T6.2** **جدول:** `sticky` header؛ أرقام `tabular-nums`؛ مبالغ **محاذاة بداية** (RTL) — لا «عشوائياً يسار/يمين».
- [ ] **T6.3** **شارة الحالة:** `default` / `secondary` / `destructive` — **مصفوفة** ثابتة (مسودة، مكتملة، آجل…).
- [ ] **T6.4** **فارغ:** «لا فواتير في الفترة المختارة» + زر «فاتورة جديدة» أو تعديل الفلتر.
- [ ] **T6.5** **ترقيم صفحات** (إن وُجد): «الصفحة 2 من 10» + أزرار واضحة.

### 5.2 نموذج الفاتورة — **إلزامي**

- [ ] **T6.6** **بنود:** جدول: منتج، كمية، سعر، إجمالي سطر — **سطر جديد** بزر `outline` واضح «إضافة بند».
- [ ] **T6.7** **بحث منتج:** combobox **سريع** — لا 500ms re-render بكل حرف (debounce).
- [ ] **T6.8** **المجاميع:** صندوق `bg-muted/40` + **إجمالي قبل وبعد** الضريبة (إن وُجد) — **سطر واحد** بلا ازدواج.
- [ ] **T6.9** **Validation:** «أضف بندًا واحدًا على الأقل» / «الكمية يجب أن تكون أكبر من صفر».

### 5.3 تفاصيل الفاتورة — **إلزامي**

- [ ] **T6.10** **رأس:** رقم + تاريخ + عميل + حالة — `h1` للرقم أو للعنوان «فاتورة مبيعات» + الرقم **بارز** جنبها.
- [ ] **T6.11** **إجراءات:** «طباعة»، «تسجيل دفع» (إن وُجد) — **primary** لأهم إجراء **بعد** الاطلاع على الحالة.
- [ ] **T6.12** **سجل دفعات** (إن وُجد): جدول صغير — لا يختلط ببنود الفاتورة.

### 5.4 طباعة A4 — **إلزامي**

- [ ] **T6.13** `InvoicePrint`: شعار/اسم الشركة (إن وُجد) + **حدود** فاصلة بين الأقسام.
- [ ] **T6.14** `page-break-inside: avoid` للصفوف (حيث يناسب) — لا **قطع** صف بند في منتصف A4.
- [ ] **T6.15** **إخفاء** تطبيق: sidebar/header **لا** تظهر في print.

### 5.5 دفع جزئي — **مهم**

- [ ] **T6.16** `InvoicePaymentDialog`: حقل مبلغ + **المتبقي** تحته `text-sm`؛ تعطيل عند `pending`.
- [ ] **T6.17** **نجاح:** toast «تم تسجيل الدفع» + تحديث واجهة.

### 5.6 عروض الأسعار — **مهم**

- [ ] **T6.18** **تحويل لـ فاتورة** (إن وُجد): زر `primary` «إصدار فاتورة» + تأكيد بسيط.
- [ ] **T6.19** **حالات:** مسودة/مُرسل/مقبول — شارات **لا تلتبس** بفواتير المبيعات.

### 5.7 «بيع جديد» vs POS — **مهم**

- [ ] **T6.20** صفحة `sales/new`: جملة توضيح `text-sm text-muted-foreground`: **«للفواتير المفصّلة وإدارة الائتمان — للبيع السريع استخدم شاشة نقطة البيع»** (يُضبط النص حسب المنتج).

## 6. نسخ جاهز

| الحالة | نص |
|--------|-----|
| لا بنود | «أضف بندًا واحدًا على الأقل» |
| دفع | «المتبقي: {مبلغ}» |
| طباعة | زر «طباعة» / «معاينة الطباعة» |

---

## 7. نعم / لا

| نعم | لا |
|-----|-----|
| ترتيب أعمدة ثابت في كل فواتير | أعمدة تتبدل بين القائمة والتفاصيل |
| صندوق ملخص مالي | أرقام مبعثرة بلا إطار |
| `ConfirmDialog` للحذف/الترحيل الحساس | تأكيد غامض «موافق؟» |

---

## 8. معايير القبول

- رحلة: قائمة → تفاصيل → طباعة **مقبولة** بصريًا.
- لا إنجليزي في واجهة المبيعات.
- Handoff لشكل **استجابة الـ API** دون اختراع UI وهمي.
