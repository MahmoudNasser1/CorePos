# خطة تحسين CorePOS بناءً على مرجع Flow POS  
**المرجع:** [flow-pos.com/ar/register](https://flow-pos.com/ar/register) (صفحة التسجيل + الهيكل العام كما ظهرت في 2026-04-25)  
**الغرض:** استخلاص **أنماط UX/UI واقتراحات منتج** قابلة للتنفيذ في CorePOS **دون** نسخ نصوص/تصميم حرفيًا (ملكية فكرية + هوية منفصلة).

---

## 1. ماذا يفعل مرجع Flow بشكل قوي؟ (تحليل مُختصر)

| البعد | ماذا لاحظنا؟ | لماذا يُحسّن التحويل والثقة؟ |
|--------|----------------|------------------------------|
| **هيكل الصفحة** | تخطيط **مقسوم**: جنبٌ يشرح **قيمة المنتج** (فقرة + **شبكة ميزات** 12 بندًا مع أيقونات اختيار) + جنب/عمود **نموذج تسجيل** داخل **بطاقة** بيضاء بظل خفيف. | المستخدم **يعرف ماذا يشتري** قبل إدخال بيانات — يقلل الارتداد. |
| **نموذج التسجيل** | حقول **مترابطة بسياق متجر/شركة**: اسم المحل، العنوان، **الدولة** (قائمة واسعة للدول العربية)، **الهاتف** مع اختيار دولة/علم، **بريد مدير النظام**، **رابط المتجر** كحقل **مركّب** (بادئة نطاق ثابتة + **slug** يختاره المستخدم)، **كلمة مرور** مع **إظهار/إخفاء**، **كود إحالة اختياري**. | يرسّخ فكرة «نظام + متجر سحابي» وليس «حساب بريد فقط». |
| **وضوح إلزامي** | الحقول الإلزامية مُعلّمة بـ **\*** (حمراء في الواجهة البصرية). | تقليل أخطاء الإرسال والدعم. |
| **عنوان الصفحة (SEO)** | `إنشاء حساب جديد | جرب مجاناً | الماركة` — يوضح **العرض** (تجربة مجانية) وليس العنوان فقط. | يدعم المشاركة والتبويبات والنتائج. |
| **تنقل تسويقي** | روابط: عن المنتج، الباقات، أسئلة شائعة، شرح، دخول، CTA تسجيل. | المستخدم لا يحسّ بالحبس في صفحة دون رؤية التسعير/الثقة. |
| **تذييل** | هاتف، بريد، **شروط/خصوصية**، **واتساب**. | طبقة ثقة + قنوات دعم. |
| **دعم سريع** | زر/أيقونة **واتساب** عائم (ظاهر بصريًا). | تقليل احتكاك «عندي مشكلة في التسجيل». |
| **CTA** | زر **«ابدأ الاستخدام مجاناً»** بارز؛ في لقطة التصفح كان **معطّلًا** حتى اكتمال الشروط (افتراض واجهة). | يمنع إرسالًا ناقصًا (يُنذر لسلوك جيد). |

> **ملاحظة:** الصفحة **RTL** بالكامل، مساحات بيضاء واسعة، لوحة ألوان هادئة (أبيض/رمادي + أزرق للهوية) — يتماشى مع [00-design-principles.md](./ui-ux-modules/00-design-principles.md) (بساطة + هرمية).

## 2. أين نحن في CorePOS اليوم؟ (فجوة وظيفيّة/تجريبية)

| الجانب | Flow (مرجع) | CorePOS (حاليًا تقريبًا) |
|--------|---------------|--------------------------|
| **تسجيل** | شركة + عنوان + دولة + هاتف + بريد مدير + **نطاق/رابط متجر** + كود إحالة | [register/page.tsx](../../src/app/(auth)/register/page.tsx): **اسم كامل + بريد + كلمة مرور + تأكيد** فقط |
| **صفحة التسجيل كبضاعة** | ميزات مُعَدّة في الشبكة على نفس الشاشة | [marketing/page](../../src/app/(marketing)/page.tsx): **landing منفصلة** عن `/register` — تمرين جيد لكن **رحلة «من إعلان إلى حساب+شركة»** أقل ارتباطًا بمرحلة One-page لدى Flow |
| **ثقة وقانون** | تذييل: شروط، خصوصية، واتساب | راجع [Footer](../../src/components/marketing/Footer.tsx) إن وُجد — قد تكون أقل اكتمالًا من التذييل المرجعي (يُقترح **معادلة** تغطية الروابط) |
| **هاتف بدلالة دولة** | حقل هاتف + اختيار دولة/علم | غالبًا **غير موجود** في نموذج التسجيل الحالي |
| **نطاق فرعي (slug)** | جزء من التسجيل (SaaS كلاسيكي) | **يتطلب** دعم backend/استضافة/DNS — خارج نطاق «تعديل واجهة فقط» إن لم تكن مُهيّأة |
| **كود إحالة** | اختياري | اختياري = **ميزة منفصلة** (تتبع، مكافآت) — قرار منتج |

**الاستنتاج:** أقرب شيء يمكن الاقتراب منه **بسرعة** هو **طبقة UI/UX + نسخة الصفحة** (تخطيط مقسوم، ميزات، بند قانوني، واتساب). أما **حقول الشركة والنطاق والهاتف الدولي** فهي **مرحلة منتج+API** بمجموعة واضحة.

## 3. مبادئ نأخذها من المرجع (دون تقليد بصري حرفي)

1. **صفحة التسجيل = صفحة بيع**: نص قيمة قصير + **قائمة ميزات** قابلة للمسح (3 أعمدة على الديسكتوب، عمود واحد على الموبايل) بجانب أو فوق النموذج.  
2. **النموذج داخل «كرت» واحد** بعنوان واضح مثل: «أنشئ حسابك لنظامك» (صياغة **خاصة بك**).  
3. **تمييز الحقول الإلزامية** (`*` أو `required` + `aria-required`) + رسائل ميدانية.  
4. **كلمة المرور:** زر إظهار/إخفاء (موجودة كأفضل ممارسة في السوق).  
5. **رابط واضح:** «لديك حساب؟ تسجيل الدخول».  
6. **قنوات دعم:** واتساب/بريد في **تذييل موحّد** على (تسجيل/دخول/لاندنغ).  
7. **ميتاداتا:** `title` يذكر **التجربة/العرض** إن كان المنتج يقدّم تجربة مجانية (يتوافق مع `metadata` في `layout` إن وُسّع نطاقه لمسارات `(auth)`).

## 4. خطة مراحل تنفيذ (للمشروع)

### المرحلة A — **واجهة + محتوى** (منخفض الاعتماد على الـ API)

| # | المهمة | المخرج | ملفات محتملة |
|---|--------|--------|---------------|
| A1 | إعادة تخطيط `/register` إلى **شبكة**: `lg:grid-cols-2` — عمود 1: قيمة + ميزات (نصوص **CorePOS** من [marketing](../../src/app/(marketing)/page.tsx) أو مختصَرة) | صفحة تسجيل «أقرب لـ flow» | `(auth)/register/page.tsx` + مكوّن `RegisterHero` اختياري |
| A2 | مكوّن **ميزات** (12 بند أو 9) مع `Check`/`CheckCircle2` — **لا ننسخ** جمل Flow؛ نعيد كتابتها من ميزاتكم الفعلية | نفس المبدأ: شبكة 3×4 | `src/components/auth/` أو `marketing/` |
| A3 | لف النموذج بـ `Card` + `CardHeader` (عنوان + وصف `text-sm`) | فصل بصري أوضح | shadcn `Card` |
| A4 | **إظهار/إخفاء** كلمة المرور + (اختياري) لحقل التأكيد | A11y + `aria-pressed` على الزر | نفس الصفحة |
| A5 | **تذييل** موحّد على `(auth)/layout` (إن لم يكن): واتساب (رابط `wa.me/...` من env)، بريد، **شروط/خصوصية** (مسارات فعلية أو `#` + TODO) | اتساق مع [02-auth plan](./ui-ux-modules/02-auth-onboarding-billing.md) | `(auth)/layout.tsx` |
| A6 | **Metadata** لصفحة التسجيل: `title` / `description` عربية تشمل **عرض** المنتج (مثال: «تجريبي» إن ينطبق) | SEO وعلامة تبويب واضحة | `register/page` أو `layout` |

**معيار جاهزية A:** `npm run lint` + `tsc`؛ **لا** تغيير عقد `/auth/register` حتى تُنفّذ B.

---

### المرحلة B — **نموذج تسجيل أغنى (يتطلب اتفاق API)**

قبل البرمجة: **عرّف** ما إذا كان التسجيل ينشئ `User` فقط أم `Company` + `User` + (اختياري) `Tenant slug`.

| # | حقل (مقاربة Flow) | ملاحظة تقنية / منتج |
|---|-------------------|------------------------|
| B1 | **اسم الشركة / المحل** | يغذي `company.name` عند الـ onboarding |
| B2 | **العنوان** (اختياري/إلزامي حسبكم) | حقل نصي |
| B3 | **الدولة** | `Select` من قائمة (ISO) — ابدأ بـ 8–10 دول + «أخرى» |
| B4 | **الهاتف** | `input` + `Select` مفتاح دولة (بدون مكتبة ثقيلة في MVP) أو `libphonenumber` **لاحقًا** |
| B5 | **بريد مدير النظام** | يساوي `email` الحالي أو `ownerEmail` |
| B6 | **رابط المتجر / الـ slug** | يتطلب: **تفرّد** + سياسة أحرف لاتينية؛ ربط لاحق بـ subdomains (DevOps) |
| B7 | **كود إحالة** | تخزين + لاحقًا تتبع (جدول `referrals`) — يمكن **حقل UI** بدون منطق لاحقًا = hidden أو disabled |

**معيار B:** عقد `POST /auth/register` (أو endpoint موحّد) **موثّق** في [HANDOFFS](../agent_reports/HANDOFFS.md) + migrations إن لزم.

---

### المرحلة C — **لاندنغ + رحلة كاملة (تسويق)**

| # | المهام |
|---|--------|
| C1 | محاذاة **hero** لديك مع رسالة «سحابي + فروع + دعم» (من Flow نستعير **الفكرة** فقط) |
| C2 | **قسم أسعار** يوضح ماذا داخل **التجريب** (لديكم شارة «معاينة» — يمكن **توضيح** الحدود بجدول بسيط) |
| C3 | روابط: `/register` + `/login` + `#pricing` + مساعدة/دعم (حتى واتساب) **في كل صفحات دخول التحويل** |

## 6. ما لا نفعله (حدود / مخاطر)

- **عدم** نسخ **نصوص/شعارات/ألوان** Flow حرفيًا — مرجع **تجربة** فقط.  
- **تعدد النطاقات (multi-tenant subdomain)** يلمس **بنية backend وDNS** — ليس «تحسين CSS» وحد.  
- الالتزام بـ [docs/CONTEXT.md](../CONTEXT.md): MVP، **لا Paymob** في الـ MVP إن كانت سياستكم كذلك — صفحات **الباقات** عندكم قد تكون **يدوية/واتساب**؛ اذكروا ذلك **بصراحة** في UI لتطابق التوقع.

## 8. الخلاصة (جملة واحدة)

**استلهم من Flow «بيع الصفحة» (قيمة + ميزات + نموذج مُحكَم + ثقة + دعم)، وابنِ في CorePOS **نفس المنطق** بمحتوى وهوية خاصة بك، مع توسيع الحقول تدريجيًا بما يتوافق مع **عقد الـ API** — وليس بمحاكاة سطحية لصفحة المنافس.**

## 9. ما تم تنفيذه في المشروع (2026-04-25)

> **تنفيذ أفكار المرجع بشكل مختلف** — تخطيط bento للميزات، بطاقات shadcn، تدرج خلفية هادئ، **بدون** تقليد تخطيط/هوية Flow.

| الفكرة من المرجع | التنفيذ في CorePOS |
|-------------------|---------------------|
| صفحة تسجيل = قيمة + نموذج | `RegisterValuePanel` (شبكة 2×3 كروت بأيقونات) + `Card` للنموذج |
| شريط علوي + روابط | `AuthPageShell`: شعار + المميزات/الباقات/دخول |
| تذييل ثقة + واتساب | `AuthMarketingFooter` + `NEXT_PUBLIC_SUPPORT_WHATSAPP` (اختياري) |
| إظهار كلمة المرور | `AuthPasswordField` — تسجيل + دخول |
| ميتاداتا صفحة تسجيل/دخول | `register/layout.tsx`, `login/layout.tsx`, `forgot-password/layout.tsx` |
| مسارات auth موحّدة | `(auth)/layout.tsx` يلفّ كل صفحات المصادقة |

**ملفات جديدة رئيسية:** `src/components/auth/AuthPageShell.tsx`, `AuthMarketingFooter.tsx`, `RegisterValuePanel.tsx`, `AuthPasswordField.tsx` — وتعديل `register` / `login` / `forgot-password` pages.

**لاحقًا (نفس الاتجاه):** صفحتان `/terms` و `/privacy` (نصوص مبدئية + مراجعة قانونية) وروابط فعّالة من `AuthMarketingFooter` و`Footer` التسويقي؛ وملف **`.env.example`** يوضح `NEXT_PUBLIC_SUPPORT_WHATSAPP` لظهور واتساب في تذييل المصادقة.
